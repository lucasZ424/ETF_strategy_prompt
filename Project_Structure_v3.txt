1) Core Design Principles (Non-negotiable)

- Contract-first:
  Keep one canonical data contract across config, schema validation, feature
  engineering, training, inference, and backtest.

- No-lookahead:
  All cross-market features must be strictly lagged and aligned to China market
  decision timestamps.

- Reproducibility:
  Every training run must be traceable (config snapshot, compact feature list,
  params, metrics, signal thresholds, version tag).

- Config-driven behavior:
  No hardcoded symbol assumptions in feature generation, threshold logic,
  portfolio logic, or inference behavior.

- Pilot honesty:
  This is paper-trading / research support only, not live investment advice.


2) Data and Modeling Objective (Updated)

- Forecast target:
  Forward 5-day log return.
- Objective rationale:
  Move the pilot from one-step sensitivity toward short-horizon tradability,
  lower turnover, and better post-cost robustness on unseen ETFs.
- Core market:
  Representative China ETFs.
- Cross-market context:
  US/EU ETFs and macro risk proxies with strict lag alignment.
- Baseline model path:
  1. XGBoost first (primary baseline)
  2. LSTM optional after XGBoost baseline is stable
- Prediction-to-trade conversion:
  Use volatility-scaled thresholded no-trade-region logic instead of trading on
  every predicted sign.
- Threshold philosophy:
  This project targets an advanced research-grade pilot rather than a simple
  course-level demo, so signal activation thresholds will be volatility-scaled
  to improve transferability across ETFs with different realized volatility.

Representative universe for current pilot:
- China core: 510050.SS, 510300.SS, 510500.SS, 159915.SZ, 588000.SS
- Cross-market context: SPY, QQQ, IEUR


3) Pilot Architecture

Data plane:
1. Fetch raw data from Yahoo Finance into `data/raw/`.
2. Clean, align, and engineer features into `data/processed/features_v2.*`.
3. Build target labels as forward 5-day log return.
4. Train/evaluate models and output versioned artifacts to `models/` and
   `outputs/<version>/`.

Service plane:
1. Backend API (FastAPI) endpoints for health, model metadata, and prediction.
2. Frontend dashboard for model status, prediction table, and strategy prompt
   display.

Control plane:
1. Config management (`configs/*.toml`).
2. Schema validation (`src/data/schema.py`).
3. Tests and quality gates (unit + integration).
4. Run traceability with consistent version tags.


4) Development Phases with Exit Criteria

Phase 0: Pipeline Stabilization (must finish first)

Tasks:
1. Fix fetch script runtime issues and data summary generation.
2. Unify column naming and target basis across config/schema/features.
3. Remove hardcoded cross-market symbol assumptions from feature logic.
4. Align config templates with parser behavior (no silent ignored fields).
5. Refresh README commands to match actual workflow.

Exit criteria:
- A fresh clone can run fetch -> pipeline -> train -> plot without code edits.


Phase 1: Data Retrieval and Preprocessing

Tasks:
1. Fetch and store China + cross-market ETF history from Yahoo Finance.
2. Normalize yfinance output schema to stable internal columns.
3. Data cleaning policy:
   - China core ETFs: drop invalid/NaN rows in core columns.
   - Cross-market context: forward-fill and align by strict lag rule.
4. Build feature set (see `ETF_feature_engineering_summary_v2.txt`).
5. Apply feature selection in two stages for non-cross-market features:
   - Stage A: Pearson correlation filter to remove highly redundant features.
   - Stage B: XGBoost importance ranking on validation results to keep the most
     stable and decision-relevant features.
6. Keep cross-market and macro context block unchanged during compact-feature
   redesign to preserve external regime information.

Exit criteria:
- Deterministic feature dataset generated with documented schema, compact
  feature list, and no leakage.


Phase 2: Data Split and Training Pipeline

Tasks:
1. Chronological train/val/test split (date-based, no leakage).
2. Train XGBoost baseline with Bayesian tuning + pruning.
3. Save model bundle and training metadata.
4. Evaluate with prediction metrics:
   - MSE, RMSE, MAE, R2, directional accuracy
5. Calibrate volatility-scaled signal thresholds on the validation set rather
   than fixing them by intuition.
6. Save threshold calibration outputs together with model artifacts.

Volatility-scaled threshold implementation logic:
- Let `pred_t` be the predicted forward 5-day log return at time t.
- Let `vol_t` be a recent realized volatility estimate for the same ETF
  (example: rolling std or realized vol over a fixed lookback window).
- Define upper/lower trade thresholds as:
    upper_t = +k * vol_t
    lower_t = -k * vol_t
- Signal rule:
    if pred_t > upper_t  -> long / buy signal
    if pred_t < lower_t  -> sell / short-or-flat signal (depending on mandate)
    else                 -> no-trade / hold current state
- Candidate k grid for validation calibration:
    0.25, 0.50, 0.75, 1.00
- Calibration target:
  Prefer the k that produces the best post-cost Sharpe subject to acceptable
  turnover, sufficient trade count, and stable behavior across ETFs.

Exit criteria:
- Reproducible training run with saved model, metrics, threshold calibration
  results, and prediction outputs.


Phase 3: Local Strategy Integrity Test (pilot backtest layer)

Tasks:
1. Convert predictions to volatility-scaled threshold-based buy/sell/flat
   signals with a no-trade region.
2. Run local backtest simulation with explicit assumptions.
3. Report:
   - Equity curve
   - Return distribution
   - Sharpe, CAGR, max drawdown
4. Add transaction cost and slippage assumptions in every report.
5. Add the following decision-quality and turnover diagnostics:

+-------------------------------+-----------------------------------------------+
| Metric                        | What it measures                              |
+-------------------------------+-----------------------------------------------+
| Turnover                      | How much the portfolio/position changes over  |
|                               | time; direct proxy for cost pressure.         |
+-------------------------------+-----------------------------------------------+
| Average holding period        | Mean duration that a position remains open;   |
|                               | checks whether the strategy is too reactive.  |
+-------------------------------+-----------------------------------------------+
| Number of trades              | Total count of executed entries/exits; sanity |
|                               | check on strategy activity level.             |
+-------------------------------+-----------------------------------------------+
| Post-cost Sharpe              | Risk-adjusted return after commissions and    |
|                               | slippage; primary tradability metric.         |
+-------------------------------+-----------------------------------------------+
| Hit rate conditional on trade | Directional accuracy only on timestamps where |
| only                          | the strategy actually executed a trade.       |
+-------------------------------+-----------------------------------------------+
| Validation threshold          | Threshold-selection result from validation;   |
| calibration result            | links signal strength to economic outcomes.   |
+-------------------------------+-----------------------------------------------+

Exit criteria:
- Strategy report is reproducible, assumptions are explicit in outputs, and the
  backtest includes both prediction metrics and post-cost trading diagnostics.


Phase 4: Backend API Pilot

Tasks:
1. Build FastAPI service wrapping current model inference.
2. Expose endpoints:
   - `GET /health`
   - `GET /model/latest`
   - `POST /predict`
3. Enforce schema checks before inference:
   - missing columns
   - insufficient history
   - invalid date order
4. Return both model prediction and threshold-derived strategy state, if the
   required volatility context is available.

Exit criteria:
- Deterministic local inference API working with sample payloads.


Phase 5: Frontend Dashboard Pilot

Tasks:
1. Build lightweight dashboard (Grafana or simple web UI).
2. Minimum panels:
   - Recent price trend (if available)
   - Top ETF predicted returns table
   - User-selected ETF prediction + strategy prompt
   - Model version and last retrain time
   - Current volatility-scaled threshold status (inside band / triggered)

Exit criteria:
- End-to-end local demo: input -> backend prediction -> frontend display.


Phase 6: Quality and Automation

Tasks:
1. Add integration tests for pipeline + training + inference API smoke run.
2. Add weekly retrain job with logging and failure alerts.
3. Add CI checks for lint/tests before merge.

Exit criteria:
- Regressions are caught automatically; weekly retrain is auditable.


5) Open-Universe Inference Design (for deployment)

Rules:
1. Keep feature schema ID-agnostic (no ETF identity as core predictor).
2. Require fixed input columns and minimum lookback length (example: 60 days).
3. If insufficient history, return `not_enough_history` (no forced prediction).
4. Keep cross-market features strictly lagged to China decision time.
5. Validate generalization with leave-one-ETF-out tests to simulate unseen ETFs.
6. Preserve threshold logic as config-driven so per-run calibration results can
   be frozen and reused at inference time.


6) Portfolio Optimization Extension (planned next-stage integration)

Positioning:
- Portfolio optimization is not treated as a fully independent module.
- It will sit downstream of the prediction pipeline and consume model outputs,
  regime signals, and risk estimates to translate predictions into weights.

Planned role in the architecture:
1. Prediction pipeline produces forward 5-day return forecasts and trade states.
2. Regime layer classifies broad market condition (example: bull / bear).
3. Portfolio layer selects allocation style conditional on regime.

Illustrative regime-dependent styles under consideration:
- Bear regime: Min Variance / Mean-Variance defensive allocation.
- Bull regime: Max Sharpe / leveraged Max Sharpe style allocation.

Constraint:
- This extension is planned, but not yet part of the current baseline pilot.
- Baseline pilot must first prove that the prediction layer is stable,
  reproducible, and post-cost sensible.


7) Deferred / Future Upgrades (not in current baseline pilot scope)

2. Rich risk management and stress testing framework.
3. Reinforcement learning for long-horizon policy simulation.
4. Full website migration with complex interaction workflows.
5. Optional AI assistant for strategy explanation and market sentiment context.
