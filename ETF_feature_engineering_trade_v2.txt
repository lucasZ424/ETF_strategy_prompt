ETF Feature Engineering - Trade / Alpha-Gate-Regime Pipeline
=============================================================

Scope
-----
This file is exclusive to the trading branch.
It defines the shared feature contract for:
- alpha regression
- ternary triple-barrier gate
- regime model for optimizer-policy selection
- downstream covariance engine and CPLEX optimizer

This file intentionally excludes dashboard-specific raw close-price forecast
design. It only covers the trading branch and the datasets needed by the
trading models.

Trading targets
---------------
Primary alpha target:
    daily-estimated 10-day forward adjusted log return

Conceptually:
    y_alpha_t = log(AdjClose_{t+10} / AdjClose_t)

Auxiliary gate target:
    triple-barrier ternary label in {-1, 0, +1}

Auxiliary regime target:
    market-state label used to select optimizer policy, for example:
    aggressive / balanced / defensive

Risk engine note
----------------
The covariance engine is not a supervised prediction target.
It consumes adjusted log-return history derived from Adj Close prices across the
ETF universe and produces risk inputs such as covariance and volatility for the
CPLEX optimizer.

Why Adj Close is required
-------------------------
All trading-branch features involving closing-price dynamics must use Adj Close,
not raw Close.

Reason:
1. The trading branch is meant to learn economically meaningful ETF return
   behavior.
2. Adj Close preserves split/dividend-adjusted continuity.
3. This reduces artificial distortions around corporate-action events and makes
   return-based alpha, gate, and risk estimation more stable.

Dataset contract from one builder
---------------------------------
One shared feature builder should compute the full trading feature backbone once
and then emit four downstream views:

1. alpha dataset:
   ETF-level rows keyed by [date, symbol], filtered to features tagged `alpha`
2. gate dataset:
   ETF-level rows keyed by [date, symbol], filtered to features tagged `gate`
3. regime dataset:
   date-level rows keyed by [date], built by aggregating features tagged
   `regime` plus regime-only market-state features
4. covariance-engine inputs:
   adjusted log-return matrix by [date x symbol], derived directly from Adj
   Close and not subjected to feature screening

Usage tags
----------
Each feature carries explicit usage tags:
- `alpha`  = candidate input to the alpha regressor
- `gate`   = candidate input to the tradability gate
- `regime` = candidate input to the regime dataset

Important note:
- For ETF-level features, the `regime` tag means the feature may be aggregated
  to date-level for the regime dataset.
- Not every tagged feature must survive feature selection; the tags define the
  eligible pool.

Notation
--------
Let:
- A_t = Adj Close at date t
- O_t, H_t, L_t = Open, High, Low
- V_t = Volume
- eps = small constant

Covariance-engine inputs
------------------------
The covariance engine does not use the screened model-feature subsets above.
It should consume:
- daily adjusted log-return matrix across the ETF universe
- rolling return window consistent with the optimizer horizon
- optional EWMA or shrinkage estimator for covariance stability

This is critical:
- Adj Close is the correct source data.
- Portfolio optimization should use return and covariance inputs derived from
  Adj Close, not raw price levels.

Feature selection logic
-----------------------
A. Shared builder
- Compute the full feature backbone once.
- Emit alpha and gate datasets as ETF-level views filtered by usage tags.
- Emit regime dataset as a date-level aggregated view filtered by usage tags.

B. Alpha and gate selection
- Apply Pearson correlation filter to non-cross-market tabular blocks in the
  training set only.
- Apply XGBoost importance ranking separately for alpha and gate candidate pools.
- Preserve cross-market / macro block unless validation proves otherwise.

C. Regime selection
- Perform feature selection on the date-level regime dataset only.
- Respect chronological validation by regime date, not pooled ETF rows.
- Prefer compact market-state features over high-dimensional ETF-specific inputs.

D. Covariance engine
- No Pearson / importance screening.
- Risk estimation quality depends on the return matrix and estimator choice, not
  on the alpha / gate / regime feature subsets.

E. Horizon alignment rule
- Since the alpha target is a 10-day forward return, at least one momentum
  feature and one mean-reversion feature should exist at or near the 10-day
  scale.
- This is why ret10_adj, zscore10_adj, and ret10_over_rv10 remain mandatory
  baseline considerations.

Frozen decisions reflected here
-------------------------------
1. Trading branch uses one shared builder with multiple dataset views.
2. Primary trading target is forward 10-day adjusted log return.
3. Ternary triple-barrier gate is retained as the tradability model.
4. Regime model is required because optimizer policy depends on market state.
5. Covariance engine consumes adjusted-return history, not engineered alpha features.
6. Portfolio weights are produced by CPLEX, not pure top-rank selection.
7. Cross-market block is preserved unless later validation proves otherwise.
