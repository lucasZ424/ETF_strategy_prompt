Issue Log (Backtest and Generalization)
Date: 2026-03-06

Issue 2 (Critical): Target and backtest leg mismatch
- Current model target is gap-style (prev close -> next open), but backtest executes and evaluates open -> close.
- This means prediction objective and trading PnL objective are not aligned.
- Risk: inflated or misleading strategy validity.

Issue 3 (High): Threshold optimization objective mismatch
- Signal thresholds are tuned on directional accuracy of target, then reused for trading PnL.
- If the backtest return leg differs from target leg, threshold search objective is mis-specified.
- Risk: thresholds look statistically good but are suboptimal for realized PnL.

Issue 4 (High): Target and schema contract inconsistency
- Config says open_to_prev_close_log_return, but implementation uses adj_close in denominator.
- Open-universe config uses "adj close" while pipeline uses "adj_close".
- Risk: silent drift between intended target, actual target, and inference schema.

Issue 5 (High): Backtest metrics crash on empty-overlap edge case
- Engine can return empty equity curve when no overlapping bar/forecast dates.
- Metric function indexes first element of equity series without guarding empty input.
- Risk: runtime failure in valid edge cases or bad data windows.

Issue 6 (Medium): Cross-market feature logic still partly hardcoded
- cross_symbols argument is present but not used in cross-market alignment path.
- Relative-return generation is hardcoded to spy/qqq/ieur.
- Risk: reduced open-universe flexibility and hidden assumptions.

Issue 7 (Medium): Backtest interface contract drift
- Protocol declares run(..., config: BacktestConfig) but engine run(...) uses scalar params.
- BacktestConfig exists in both interface and config modules with different fields.
- Risk: maintenance friction and type/usage inconsistency.

Issue 8 (Medium): Test coverage does not include backtest modules
- Existing tests only cover schema validation.
- No tests for engine, strategy sizing, metric computation, and run_backtest integration.
- Risk: regressions in newly added backtest path can pass unnoticed.

Sanity Test Plan (Unseen ETF Generalization)

Goal:
- Validate whether pooled model trained on core 5 ETFs can produce useful signals for non-core ETFs.

Plan:
1) Freeze model and config
- Use one fixed trained model artifact (no retraining, no retuning).
- Save exact config and feature list used.

2) Select unseen ETF(s)
- Start with 1 non-core ETF for initial sanity check.
- Then expand to 3-5 non-core ETFs for robustness.

3) Enforce inference contract
- Same feature schema and preprocessing as training pipeline.
- Minimum lookback rule (example: >= 60 rows).
- If missing columns/history, return explicit failure state (do not force predict).

4) Run prediction and backtest
- Generate daily predictions/signals for test window.
- Run backtest with same fees/slippage assumptions as core benchmark.

5) Benchmark against simple baselines
- Always-flat
- Always-long
- Naive previous-gap style baseline
- Compare return, drawdown, Sharpe, trade count, turnover.

6) Evaluate stability
- Check if performance is consistent across unseen ETFs.
- Reject conclusions from a single ETF result.

7) Acceptance gate for pilot
- No runtime/schema failures.
- Performance better than naive baselines on majority of unseen ETFs.
- Drawdown and turnover remain within predefined pilot limits.

Notes:
- Regime modeling is intentionally out of scope for this pilot.
- This sanity test is for generalization and pipeline reliability, not production readiness.
