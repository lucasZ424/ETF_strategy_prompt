ETF Feature Engineering — Dashboard Forecast Pipeline
=====================================================

Scope
-----
This file is exclusive to the dashboard forecast branch.
It is intentionally separate from the trade feature file so that the investor-
facing close-price forecast logic stays clean and does not inherit ranking- or
gating-specific design choices.

Dashboard objective
-------------------
- One shared model artifact
- Multi-horizon outputs for 1d / 3d / 5d forecast display
- Official displayed forecast target = raw Close price

Important target convention
---------------------------
The dashboard displays raw Close forecasts, but the model should preferably be
trained on horizon-specific log-return-to-close targets and then converted back
to raw Close for display.

Recommended internal targets:
    y1 = log(Close_{t+1} / Close_t)
    y3 = log(Close_{t+3} / Close_t)
    y5 = log(Close_{t+5} / Close_t)

Then displayed forecast:
    Close_hat_{t+h} = Close_t * exp(yhat_h)

Reason:
- this preserves raw Close as the frontend output
- but gives a more stable and scalable supervised learning target than directly
  regressing raw Close across ETFs with different price levels

Design structure
----------------
This file is divided into:
1. Shared dashboard features
2. Dashboard-specific features

"Shared" here means reusable within all 1d / 3d / 5d dashboard horizons.
It does NOT mean shared with the trade file. Trade-specific design lives in the
separate trade feature file.

Notation
--------
Let:
- C_t       = raw Close at date t
- H_t, L_t  = High, Low
- O_t       = Open
- V_t       = Volume
- eps       = small constant for numerical safety

Why "Log Close" is included here
---------------------------------
In the dashboard branch, "Log Close" is used explicitly as:

    log_close = log(Close)

Necessity:
1. It stabilizes the price-level scale across ETFs.
2. It makes short-horizon trend shape easier to represent.
3. It allows one shared model to learn 1d / 3d / 5d movements without being too
   dominated by nominal price level differences.
4. It is most useful as a primitive for lag, slope, and return construction,
   not as a standalone magical predictor.

Dashboard-specific feature notes
--------------------------------
1. This branch uses raw Close because the frontend is supposed to show the
   visible market price investors recognize.
2. Unlike the trade branch, this branch does not include ranking-only features
   such as cross-sectional return z-scores or ternary gate support variables.
3. The dashboard branch should remain horizon-agnostic enough for one shared
   1d / 3d / 5d model, so highly specialized single-horizon features should be
   introduced cautiously.

Feature selection implementation logic
--------------------------------------
A. Pearson correlation filter
- Apply within the dashboard feature set only.
- Recommended start threshold: |corr| >= 0.90.
- Remove one of each highly redundant pair based on interpretability and
  horizon-wise validation benefit.

B. Shared-model usefulness check
- Even though one shared model is used for 1d / 3d / 5d outputs, check whether
  a feature helps all horizons or mainly one horizon.
- Keep features that are stable enough across the multi-horizon task.

C. Close-output rule
- Raw Close is the official displayed forecast quantity.
- Internal modeling via log-return-to-close is recommended, but final output is
  always converted back to raw Close.

Frozen decisions reflected here
-------------------------------
1. Dashboard branch is separated from the trade feature file.
2. Dashboard uses raw Close, not Adj Close, for output and close-based features.
3. One shared model artifact is preferred for 1d / 3d / 5d forecasts.
4. Separate horizon-specific models are postponed unless the shared model proves inadequate.
